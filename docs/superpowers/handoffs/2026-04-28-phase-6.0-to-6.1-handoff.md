## Carry-forward recommendations from Phase 6.0 reviews

These were flagged as **Important** during code-quality reviews but not
blocking for 6.0 ship. Fold them into Phase 6.1 work where natural.

### 1. `OpenRouterClient.complete()` — KeyError on malformed 200 response

`tasks/openrouter_client.py:163` accesses `data["choices"][0]["message"]
["content"]` without protection. If OpenRouter returns 200 with `{"error":
{"message": "..."}}` (provider-routing failures do this) or a streaming
response leaks through, callers get raw `KeyError` instead of `OpenRouterError`.
This breaks the docstring contract.

**Fix in Phase 6.1 extractor**: wrap LLM-response parsing in
`try/except (KeyError, IndexError, TypeError)` and re-raise as
`OpenRouterError` (or new `ExtractionError`). Or fix at the client level —
either is acceptable.

### 2. `Task.label_names` mutable-default isolation has no test

`tests/test_tasks_schema.py:94` — `test_task_label_ids_default_is_independent_per_instance`
only mutates `label_ids`. Mirror it for `label_names`:

```python
a.label_names.append("bug")
assert b.label_names == []
```

One-line addition. Without it, a future refactor could silently change
`label_names = field(default_factory=list)` to `= []` and tests would still
pass.

### 3. `OpenRouterClient.validate_key()` Timeout message asymmetry

`tasks/openrouter_client.py:81` — `validate_key`'s Timeout error reads
`"Таймаут подключения к OpenRouter"` (no timeout value), while `complete()`
at line 144 reads `f"Таймаут OpenRouter (>{timeout}s)"` (interpolated). When
debugging a hang, the value is useful. Suggested fix:

```python
raise OpenRouterError("Таймаут подключения к OpenRouter (>10s)") from e
```

### 4. `SettingsDialog.grab_set()` — defensive hardening

`ui/dialogs/settings.py:37` calls `self.grab_set()` with no `try/except` and
no `wm_protocol("WM_DELETE_WINDOW", ...)` registration. **This is the bug we
hit during the bug-fix session at the start of the previous chat** — the
"Audio Transcriber не реагирует" symptom. If construction of any
`_build_*_section` ever raises, the grab is held and the app appears frozen.

Phase 6.0 added two more sections (`_build_openrouter_section`,
`_build_linear_section`), each with a worker-thread Validate button, so the
risk surface grew. Suggested pattern:

```python
self.title("Настройки")
self.geometry("520x680")
self.transient(parent)

# Register cleanup BEFORE grab so X-button always releases
self.protocol("WM_DELETE_WINDOW", self._safe_close)
self.grab_set()

try:
    # ... all _build_*_section calls ...
except Exception:
    self._safe_close()
    raise

def _safe_close(self):
    try:
        self.grab_release()
    except Exception:
        pass
    self.destroy()
```

This is best done as a small standalone refactor commit, NOT folded into
Phase 6.1 feature work — it touches existing code unrelated to extract/edit.

### 5. `Task.from_dict` empty-string `local_id` asymmetry

`tasks/schema.py:104` uses `local_id=d.get("local_id") or str(uuid.uuid4())`.
A `Task` with `local_id=""` serialized via `to_dict` and re-loaded would get
a fresh UUID, breaking strict round-trip. In practice no code path produces
`""` for `local_id`, so low priority. Documenting intent with a comment is
sufficient:

```python
local_id=d.get("local_id") or str(uuid.uuid4()),  # generate fresh id if absent or empty
```
